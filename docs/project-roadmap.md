# 项目未完成能力总规划与开发计划

**日期：** 2026-03-30

## 1. 文档目的

本文只保留**尚未完成**的能力规划与开发计划。

以下内容不再在本文展开：

- 已完成的旧主线迁移
- 已完成的桌面端主线切换
- 已完成的基础导图编辑能力
- 已完成的 OpenAI Compatible AI 基础接入
- 已完成的三平台基础打包链路

这份文档的职责只有两个：

1. 统一记录当前项目仍然欠缺的能力
2. 把这些未完成能力细化为可执行的开发阶段

## 2. 规划边界

当前项目后续应继续围绕下面这条主线推进：

**桌面产品可用性 -> AI 工作流深度 -> 工程可维护性 -> 生态扩展**

本规划默认以下边界保持不变：

- 桌面主线继续使用 `Vue 3 + Vite + Tauri 2`
- AI 仅保留 `OpenAI Compatible` 协议主线
- 数据优先保存在本地，云能力不是当前优先级
- 先补稳定性与工作流，再考虑平台化与高级生态

## 3. 当前未完成能力总表

### 3.1 P0：数据安全与桌面可用性

这部分必须最先补齐，否则后续功能继续增加只会放大维护风险。

- 自动保存
- 启动时恢复未保存内容
- 文档快照、历史版本、恢复点
- 文件损坏检测与恢复副本
- 配置导入导出
- 崩溃日志、运行日志、诊断包导出
- 文件权限失败、路径异常、磁盘写入失败的明确提示

### 3.2 P1：应用入口与文档工作台

当前项目已经能编辑文件，但还缺少一个真正完整的桌面应用入口。

- 编辑器空白态首页第一屏
- 最近文件工作台
- 模板入口
- 恢复上次工作区
- 新建、打开、继续编辑的统一入口
- 首页与编辑器之间的清晰状态切换

### 3.3 P1：桌面发布与升级闭环

当前打包已经可用，但产品闭环还不完整。

- 自动更新
- 发布渠道区分：开发版 / 预览版 / 正式版
- 版本元数据和更新清单规范
- 安装签名策略
- 用户数据目录规范
- 升级前备份策略
- 卸载保留数据策略说明
- macOS / Linux 真机构建与升级验证

### 3.4 P1：AI 工作流化

当前 AI 已经能用，但还没有变成真正的编辑器工作流。

- 基于当前节点生成子节点
- 基于当前分支做扩写、压缩、重写、总结
- 基于整图生成摘要、结构重组、任务拆分
- Prompt 模板系统
- 自定义 AI 模板
- AI 结果预览
- AI 结果局部应用与替换
- AI 输出差异比对后手动接受
- Provider 预设管理
- Provider 诊断、超时重试、错误定位

### 3.5 P2：编辑效率与结构化能力

这部分决定产品是否能从“能编辑”进化到“高频可用”。

- 命令面板
- 全局快捷操作中心
- 聚焦模式
- 折叠状态记忆
- 节点链接与跨文档链接
- 节点反向引用或跳转关系
- 更强的搜索与跳转能力
- 任务属性：状态、优先级、截止时间
- 任务视图或任务汇总入口

### 3.6 P2：工程收口与质量体系

这部分不是用户直接可见的功能，但会直接决定后续是否还能持续开发。

- 移除 `@vue/compat`
- 清理 `vuex`，统一到 `pinia`
- 拆分编辑页重型组件
- 收口平台层与运行时状态接口
- 继续拆分过重依赖
- 补全核心单元测试
- 增加桌面端回归测试
- 增加构建产物检查
- 固化发布前检查流程

### 3.7 P3：生态与差异化增强

这一部分不应抢占前面阶段，但需要提前留出接口设计空间。

- Markdown / OPML / 其他开放格式的进一步互通
- 模板中心
- 插件化扩展接口
- 本地知识库辅助生成
- 导图内容批处理工具
- 导图到任务系统的导出
- 多文档组织能力

## 4. 暂不优先的方向

以下方向暂不进入主线排期：

- 大规模实时协作
- 复杂账户体系
- 非 OpenAI Compatible 的多协议 AI 适配
- 先做云同步再补本地稳定性
- 低价值展示型功能堆叠

## 5. 开发阶段计划

### 阶段 1：数据安全底座

**目标**

先解决“能不能放心长期使用”的问题。

**交付项**

- 自动保存机制
- 未保存状态识别与恢复
- 文档快照与恢复点
- 文件异常提示统一化
- 崩溃日志与诊断信息导出

**完成标准**

- 用户在异常退出后可恢复最近工作内容
- 本地文件写入失败时有明确错误提示
- 关键配置与最近文件记录不会因异常退出而丢失

### 阶段 2：首页第一屏与文档入口

**目标**

把“进入应用之后做什么”从分散按钮升级为完整入口体验。

**交付项**

- 编辑器空白态首页第一屏
- 最近文件列表
- 模板入口
- 恢复上次工作区入口
- 新建 / 打开 / 继续编辑的统一操作区

**完成标准**

- 无文档时进入首页第一屏
- 有恢复上下文时可快速继续上次工作
- 新建、打开、最近文件三类入口路径清晰且一致

### 阶段 3：桌面更新与发布闭环

**目标**

把“能构建”升级为“能持续发布和升级”。

**交付项**

- GitHub Release 更新源规范
- 自动更新接入
- 更新前提示、下载进度、安装与重启
- 版本清单与渠道区分
- 用户数据目录与安装目录分离
- 三平台升级验证基线

**完成标准**

- 更新失败不影响旧版本继续使用
- 配置、最近文件、AI 配置在升级后保留
- 安装包、便携版、更新元数据命名统一

### 阶段 4：AI 工作流增强

**目标**

让 AI 真正嵌入导图编辑过程，而不是停留在入口工具层。

**交付项**

- 节点生成子节点
- 分支扩写、压缩、重写、总结
- 整图摘要与结构整理
- Prompt 模板系统
- 自定义模板
- AI 结果预览、差异比对、局部应用
- Provider 预设与诊断面板

**完成标准**

- AI 输出不再直接覆盖导图，必须可预览和选择应用方式
- 常见 AI 错误可以被定位到配置、网络、上游服务或响应结构
- 用户可基于模板重复执行高频 AI 操作

### 阶段 5：编辑器效率增强

**目标**

补齐高频使用时最影响效率的操作能力。

**交付项**

- 命令面板
- 聚焦模式
- 折叠状态记忆
- 节点链接与跳转
- 跨文档链接预留
- 任务属性与任务汇总入口
- 搜索与跳转增强

**完成标准**

- 高频操作可通过键盘快速触发
- 大型导图中可快速定位节点和任务
- 节点关系不再局限于树结构

### 阶段 6：架构收口与依赖治理

**目标**

降低后续迭代成本，避免功能越多越难改。

**交付项**

- 移除 `@vue/compat`
- 清理 `vuex`
- 完成 `pinia` 统一
- 拆分编辑页超大组件
- 收口平台接口
- 继续拆分过重依赖

**完成标准**

- 不再同时依赖两套状态管理思路
- 编辑器核心模块边界清晰
- 关键模块可以独立测试

### 阶段 7：测试、回归与发布自动化

**目标**

让“修一个功能坏一片”的风险明显下降。

**交付项**

- 核心单元测试
- 桌面文件流回归测试
- AI 配置与请求链路测试
- 三平台构建产物校验
- 发布前自动检查链路

**完成标准**

- 核心功能有基础自动化回归覆盖
- 发布前可通过标准化脚本完成验证
- 构建警告与明显回归可在发布前被拦截

### 阶段 8：生态与高级能力

**目标**

在主线稳定后，为后续差异化能力做延展。

**交付项**

- 模板中心
- 开放格式互通增强
- 插件接口预研
- 本地知识库辅助生成
- 批处理工具

**完成标准**

- 主线能力稳定后，扩展能力能通过明确边界接入
- 不为了扩展能力重新破坏主线架构

## 6. 细化执行清单

### 第一批必须立即推进

1. 定义文档脏状态、自动保存和恢复的统一状态模型
2. 落地自动保存与异常恢复
3. 增加快照与恢复点
4. 统一文件系统错误模型与提示
5. 实现首页第一屏
6. 把最近文件、模板、新建、打开整合为统一入口
7. 定义自动更新源、更新元数据和渠道规则
8. 接入更新检查、下载、安装与重启流程

### 第二批紧跟推进

9. 增加节点级和分支级 AI 工作流
10. 实现 AI 结果预览、差异比对和局部应用
11. 增加 Provider 预设、诊断和重试策略
12. 实现命令面板
13. 增加节点链接、任务属性和任务汇总入口

### 第三批工程收口

14. 移除 `@vue/compat`
15. 清理 `vuex` 并统一到 `pinia`
16. 拆分编辑页大组件和平台层
17. 继续拆分过重依赖与静态资源
18. 补齐关键单元测试和桌面回归测试

### 第四批长期增强

19. 模板中心正式化
20. 开放格式互通继续扩展
21. 插件接口与批处理工具预研
22. 本地知识库辅助生成

## 7. 执行原则

- 先修“数据安全”和“入口体验”，再继续扩功能
- 先把 AI 做成稳定工作流，再继续扩模型或协议
- 先收口架构，再做重型高级能力
- 每一阶段都要带验收标准，不能只列想法
- 每轮开发都必须同步补测试、文档和发布校验

## 8. 结论

项目后续最重要的不是继续零散加按钮，而是把下面这条链路完整做通：

**数据安全 -> 首页入口 -> 自动更新 -> AI 工作流 -> 工程收口**

只要这条主线推进到位，项目才会从“已经迁移好的桌面版思维导图工具”升级为“可长期维护的正式桌面产品”。
