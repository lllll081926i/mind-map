# Vue 3 与新 UI / 新状态管理迁移前置清单

日期：`2026-03-30`

## 1. 本轮结论

本轮不执行 Vue 3 迁移，只完成前置清单。

结论边界：

- 不升级到 `Vue 3`
- 不升级到 `vue-router 4/5`
- 不升级到 `vue-i18n 9/11`
- 不把 `Vuex 3` 切到 `Pinia` 或 `Vuex 4`
- 不把 `Element UI 2` 切到新的 Vue 3 UI 框架

## 2. 当前迁移规模事实

按 `2026-03-30` 本地代码统计：

- `app/src` 下共有 `50` 个 `.vue` 文件
- `web/src` 下共有 `50` 个 `.vue` 文件
- `app/src` 中出现 `312` 处 `<el-` 组件标签
- `web/src` 中出现 `312` 处 `<el-` 组件标签
- `app/src` 中出现 `110` 处 `mapState / mapMutations / mapActions / mapGetters`
- `web/src` 中出现 `110` 处同类 Vuex helper
- `app/src` 中出现 `36` 处 `beforeDestroy / destroyed`
- `web/src` 中出现 `36` 处同类 Vue 2 生命周期

这说明迁移不是“换几个包版本”，而是实打实的视图层和状态层重构。

## 3. 主要阻塞点

### 3.1 双入口重复面过大

`app` 和 `web` 目前仍基本保持镜像结构。

如果直接双线同时迁移，会出现：

- 两份视图层重复改造
- 两份回归测试成本
- 问题定位时难以区分“业务差异”还是“入口差异”

前置建议：

- 后续若启动 Vue 3 迁移，应先确定 `app` 为唯一主迁移线
- `web` 只保留兼容壳，或改成复用 `app` 的共享视图层

### 3.2 `Element UI 2` 使用面太广

当前 `<el-` 标签密度很高，意味着 UI 框架替换不是简单兼容层问题。

前置建议：

- 先做组件盘点
- 先决定目标 UI 框架
- 先定义兼容策略：
  - 直接替换
  - 中间适配层
  - 渐进式包裹

### 3.3 Vuex 依赖仍深

当前大量组件仍通过 Vuex helper 直接读写状态。

前置建议：

- 先梳理 store 模块边界
- 识别哪些状态适合迁到组合式 hooks
- 再决定使用 `Pinia` 还是 `Vuex 4`

### 3.4 Vue 2 生命周期与实例 API 使用仍多

当前仍广泛使用：

- `beforeDestroy`
- `destroyed`
- 事件总线模式
- 大量基于 Options API 的组件写法

前置建议：

- 先统一事件总线替代方案
- 再迁移生命周期与组合式状态

## 4. 建议的未来迁移顺序

1. 收敛双入口，明确只迁 `app` 主线。
2. 抽离共享业务能力，减少 `app/web` 重复组件。
3. 先替换状态层接口，再替换 UI 框架。
4. 再进入 `Vue 3 + Router + I18n` 联动迁移。
5. 最后处理编辑器体系与更深层 UI 行为差异。

## 5. 进入正式迁移前必须完成的准备

### 5.1 代码层

- 建立组件迁移清单
- 建立 store 模块清单
- 建立 Element UI 组件替换清单
- 建立全局实例 API 使用清单

### 5.2 验证层

- 补齐主流程冒烟：
  - 编辑
  - 主题
  - 导入导出
  - AI
  - 备注
  - 图片预览
- 至少保证 `app` 有稳定 build 和 smoke 基线

### 5.3 发布层

- 明确根目录发布只由哪条入口线负责
- 明确 `web` 在 Vue 3 时代的定位：
  - 保留
  - 冻结
  - 兼容壳
  - 下线
