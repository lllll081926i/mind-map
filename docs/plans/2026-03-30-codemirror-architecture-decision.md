# CodeMirror 架构结论

日期：`2026-03-30`

## 1. 结论

本轮不实施 `CodeMirror 5 -> 6` 迁移。

当前正式结论：

- 继续维持 `CodeMirror 5`。
- `CodeMirror 6` 只保留为后续独立迁移项目。
- 不把编辑器重构混入普通依赖升级。

## 2. 当前事实

### 2.1 现有依赖基线

- `app/package.json` 仍直接依赖 `codemirror@^5.65.21`
- `web/package.json` 仍直接依赖 `codemirror@^5.65.21`
- 两端都仍依赖 `@toast-ui/editor@^3.1.5`

### 2.2 代码证据

当前仓库没有发现以下内容：

- `@codemirror/*`
- `codemirror@6` 相关包
- 面向 CodeMirror 6 的扩展或状态管理代码

当前备注 / Markdown 编辑链仍以 `Toast UI Editor 3 + CodeMirror 5` 为基础。

## 3. 为什么本轮不迁

### 3.1 直接收益不足

- 本轮的主目标是把 Node、构建链、依赖治理和双入口兼容线稳定下来。
- 编辑器迁移不会直接解决当前的 Node 25、CLI 5、Webpack 5、根目录发布等阻塞项。

### 3.2 迁移范围不是单包升级

`CodeMirror 6` 不是补丁升级，而是架构级替换。按当前仓库形态，至少会牵动：

- 备注编辑器封装
- Markdown 编辑体验
- 样式覆盖
- 快捷键和编辑状态同步
- 懒加载边界
- 移动端输入兼容

### 3.3 现有耦合点仍在

这里是基于源码的工程判断：

- 当前备注编辑能力仍围绕 `@toast-ui/editor@3` 组织。
- 在未替换备注编辑器宿主之前，单独把 `CodeMirror 5` 抽换成 `6` 的收益很低，改动面却很大。

上面这点属于对现有代码结构的推断，不是第三方官方兼容声明。

## 4. 当前可接受方案

本阶段接受的稳定方案只有一个：

- 保持 `Toast UI Editor 3 + CodeMirror 5`

同时继续保留本轮已完成的优化：

- 备注展示侧按需加载
- 首屏重依赖拆分

## 5. 如果未来单独立项

需要先满足以下前置条件：

1. 先决定是否继续保留 `Toast UI Editor` 作为宿主。
2. 明确备注编辑是继续做 Markdown 编辑器，还是转为自研轻量编辑链。
3. 准备最小回归清单：
   - 备注打开 / 编辑 / 保存
   - 代码块高亮
   - 公式展示
   - 中英文输入法
   - 移动端输入
4. 单独评估 `CodeMirror 6` 的包体积、扩展装配和样式成本。

## 6. 阶段状态

阶段 16 以“结论文档完成”的方式关闭，不进入代码迁移实施。
