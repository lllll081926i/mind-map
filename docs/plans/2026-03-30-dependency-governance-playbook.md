# 依赖治理与长期巡检手册

日期：`2026-03-30`

## 1. 目标

把依赖升级从一次性集中爆炸，改成持续治理。

适用范围：

- `simple-mind-map`
- `app`
- `web`

## 2. 固定节奏

建议节奏：

- 每两周做一次轻巡检
- 每月做一次完整巡检
- 每个季度做一次“迁移类依赖”复盘

## 3. 巡检命令

### 3.1 版本巡检

在三个工程目录分别执行：

```bash
npm outdated
```

### 3.2 构建验证

```bash
cd simple-mind-map && npm run lint
cd app && npm run build
cd app && npm run build:library
cd web && npm run lint
cd web && npm run build
```

### 3.3 双 Node 基线验证

固定保留两条基线：

- 团队建议基线：`Node 22 LTS`
- 当前本机兼容基线：`Node 25.6.0`

每次涉及构建链或底层依赖升级时，至少保证其中一条 LTS 和当前本机版本都能完成构建验证。

## 4. 升级分流规则

### 4.1 A 类：直接升级

适用：

- 安全修复
- 小版本
- 不改变框架边界

要求：

- 直接升级
- 跑构建
- 记录结果

### 4.2 B 类：条件升级

适用：

- Markdown
- 高亮
- 样式系统
- 导入导出链
- 构建工具次级变化

要求：

- 先补冒烟清单
- 再升级
- 明确回退点

### 4.3 C 类：迁移项目

适用：

- Vue 2 -> 3
- Router / I18n / Store 主版本升级
- 编辑器体系替换
- UI 框架替换
- Webpack / CLI / 打包体系主迁移

要求：

- 单独立项
- 单独文档
- 不混入普通依赖刷新

## 5. 固定验证门禁

每次升级后至少确认：

1. `simple-mind-map` lint 结果是否有新增错误
2. `app` build 是否通过
3. `app` library build 是否通过
4. `web` lint 是否通过
5. `web` build 是否通过

同时至少覆盖一次人工冒烟：

- 新建 / 删除 / 拖拽节点
- 备注编辑与展示
- 图片预览
- 导入 `.smm/.json/.xmind/.md`
- 导出 `.smm/.json/.svg/.png/.pdf/.xmind/.md/.txt`
- AI 对话与 AI 生成

## 6. 已知基线债务

截至 `2026-03-30`，仍需长期跟踪：

- `simple-mind-map lint` 仍存在 `23` 条历史遗留错误
- `web build` 仍有产物体积 warning

这些问题不能在后续升级里被误记为“新引入回归”。

## 7. 记录要求

每一批升级至少要落两类文档：

- 路线或阶段计划
- 执行记录 / 结论记录

如果阶段是“只做评估不落代码”，也必须有结论文档，不允许只停留在口头说明。

## 8. 当前推荐顺序

后续任何新升级都按以下顺序处理：

1. `simple-mind-map`
2. `app`
3. `web`
4. 根目录发布脚本

原因：

- 核心库先稳定，入口工程才有意义
- `app` 是当前主入口
- `web` 是兼容入口，不应倒逼主线设计
